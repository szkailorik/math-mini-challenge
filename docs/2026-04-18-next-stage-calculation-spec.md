# Mini Challenge 下一阶段计算训练模块开发文档

> 状态：规划稿 v0.2（已完成一次自我迭代）
>
> 目标：在不打断现有 `Mini Challenge Advanced` 训练逻辑的前提下，为系统增加一个“下一阶段”入口，用同样的日常纸面训练方式，完成小学阶段“计算相关知识点”的系统收束与过渡。

---

## 1. 这份文档要解决什么问题

当前项目已经形成了一套稳定的高密度日训系统：

- 单页应用、本地优先、可打印、可批改、可错题回炉；
- 面向两个孩子的双画像训练；
- 每日固定输出纸面训练，系统根据批改记录调整后续题目；
- 已经覆盖整数、小数、分数、互化、方程、简算等多个高价值计算板块。

但从训练路径上看，当前系统仍然更像“高强度诊断与打底阶段”，还不是一个明确的“小学计算总收束阶段”。用户希望：

1. 现有系统先保持稳定，不重构、不打断；
2. 一个月左右后，可以从当前阶段跳到“下一阶段”；
3. 下一阶段仍沿用纸面每日训练 + 系统记录 + 自适应调训的方式；
4. 训练目标更聚焦于“小学阶段所有和计算有关的知识点闭环”；
5. 下一阶段最好能保持较高训练效率，而不是简单换一批题。

这份文档给出的不是小修小补，而是一个新的训练模块规划。

## 1.1 本文明确参考的现有系统事实

这份文档不是脱离现有项目空想出来的，而是明确建立在当前仓库真实结构之上。

我参考到的现有系统关键事实包括：

- 当前项目是 `单文件静态单页应用`，核心运行时代码集中在 `index.html`；
- 当前默认训练程序实际是 `Mini Challenge Advanced`；
- 当前页面主路径是：
  - 四张学生题卷
  - 两张答案页
  - 错题本
  - 本套批改报告
  - 知识点地图
  - 数据备份 / Gist 云同步
- 当前训练对象固定是两个学生：
  - `KAI`
  - `Lorik`
- 当前 daily paper 结构是：
  - 每个学生 2 页 question sheets
  - 全部一起形成 `打印AB四页`
- 当前缓存与生成逻辑已经稳定工作：
  - `generateOrLoadSetData`
  - `SET_CACHE_PREFIX`
  - `collectRecentQuestions`
  - `sectionShuffle`
- 当前画像存储已经稳定工作：
  - `StorageDB.cache.KAI / Lorik`
  - `weights`
  - `lastSeen`
  - `history`
  - `errorBook`
- 当前反馈闭环已经稳定工作：
  - `submitGrades`
  - 错题回炉
  - 同类变式
  - 高价值训练优先
  - 本套批改报告
- 当前打印链路已经有独立 `print sandbox`，这一点必须保留，不能为下一阶段重新推翻。

所以本文的设计原则不是“另起炉灶”，而是：

- 复用你现在这套最值钱的基础设施；
- 在这套基础设施上增加“下一阶段 program”；
- 尽量不碰已经稳定的主链路。

## 2. 先做判断：下一阶段能不能完成“小学所有与计算有关的知识点”

### 2.1 结论

可以，但要先把“范围”定义准确。

如果“完成小学所有与计算有关的知识点”指的是：

- 小学数学中 `数与运算` 为主轴的内容；
- 以及 `数量关系` 中那些以运算选择、运算顺序、单位换算、比率理解、方程建模为核心的内容；
- 目标是形成稳定、可迁移、速度与准确率兼顾的计算能力；

那么完全可以把“下一阶段”设计成一个 **小学计算收束阶段**，并且有很高的产品价值。

但如果“所有知识点”被理解成：

- 几何测量、
- 图形与空间、
- 统计与概率、
- 全部应用题类型、
- 所有建模与综合情境题，

那它就不再是“计算阶段”，而会变成另一个更大的小学数学总复习系统。那样会把阶段目标做散，不建议在这个模块里一次性承载。

### 2.2 依据

我参考了教育部《义务教育数学课程标准（2022年版）》中与小学阶段相关的内容要求和学业质量描述：

- 课程标准把小学阶段的重点放在“数与运算”“数量关系”等主线，并明确强调理解算理、掌握算法、形成运算能力与推理意识。
- 第三学段（5～6年级）的学业质量描述明确提到：学生应理解小数和分数，能进行简单的小数和分数四则运算和混合运算，感悟运算的一致性，形成数感和运算能力。

基于这些官方要求，我做出的工程化判断是：

- **可以把下一阶段定义为“小学计算闭环阶段”**；
- 但应明确边界为：`小学数与运算 + 以计算为核心的数量关系`;
- 这是对课标要求的工程化映射，不等于“完成小学数学全部内容”。

## 3. 对现有系统的判断

### 3.1 现有系统已经完成了什么

当前 `Mini Challenge Advanced` 已经具备下面这些很强的基础能力：

- `整数计算`：大数减法、借位链、乘除法结构；
- `小数计算`：小数乘除法、位值、移位、商的小数点；
- `分数计算`：分数四则、带分数、交叉约分、分数与小数桥接；
- `互化比较`：分数、小数、百分数、循环小数、基准比较；
- `方程与数量关系`：逆运算、两步方程、特殊位置未知数、比例方程；
- `简算与策略`：凑整、分配律、组合拆分、补偿、裂项等；
- `自适应逻辑`：弱项加权、遗忘拦截、错题本回炉、同类变式、高价值训练优先；
- `纸面工作流`：稳定打印、答案页、批改、归档、错题复习。

更具体地说，当前系统已经有 6 个非常成熟、值得直接复用的底座：

1. `出卷底座`
   - 基于 `generateOrLoadSetData` 生成稳定 set；
   - 已有 recent question 去重和 section 内洗牌。

2. `画像底座`
   - `StorageDB` 已经把权重、lastSeen、history、errorBook 连成闭环。

3. `批改底座`
   - `submitGrades` 已经能把答案页批改回写到画像，并触发后续训练。

4. `打印底座`
   - `printQuestionSheets` / `printAnswerSheets` 已经用独立 print sandbox 稳定运行。

5. `复盘底座`
   - 现在已有 `Set Review`、错题本、知识点地图三种复盘界面。

6. `同步底座`
   - `GistSync + StorageDB` 已经把本地优先和跨设备同步跑通。

### 3.2 现有系统还没有专门完成什么

当前系统虽然已经很强，但从“完成小学计算闭环”的角度看，还缺一层更明确的收束设计：

- 缺少一个清晰的“阶段毕业”标准；
- 缺少“当前阶段 -> 下一阶段”的正式接口；
- 缺少专门覆盖 `整数 / 小数 / 分数 / 百分数 / 比 / 比例 / 单位换算 / 估算策略 / 混合运算` 的统一收束模块；
- 缺少把“会算”进一步收束为“会选择方法、会迁移、会压缩错误率、会保持速度”的专门阶段；
- 现有训练更像“高质量日常强化”，下一阶段则应该更像“体系闭环与升段前总收束”。

所以，**不是当前系统不够好，而是它适合做 Stage 1；现在值得规划一个 Stage 2。**

## 4. 方案对比与推荐

在正式落设计前，有 3 条实现路径。

### 4.1 路径 A：直接在现有题库里继续加题

做法：

- 继续在当前 `Mini Challenge Advanced` 里扩 section；
- 增加更多题型和更多标签；
- 用当前缓存、画像、打印流直接承接。

优点：

- 实现最快；
- UI 改动最小；
- 复用现有逻辑最多。

缺点：

- 当前系统会越来越像“大杂烩”；
- 无法清楚表达“当前阶段”和“下一阶段”；
- 数据画像会把两个阶段揉在一起；
- 未来再继续加第三阶段会更难。

判断：

- 不推荐作为正式方案；
- 只适合做临时验证，不适合做产品级结构。

### 4.2 路径 B：复制一份新页面或新仓库

做法：

- 保留当前项目不动；
- 新建一个 `mini-challenge-closure` 页面或仓库；
- 把下一阶段完全独立开发。

优点：

- 技术隔离最强；
- 不会碰当前系统；
- 上线风险低。

缺点：

- 用户体验割裂；
- 数据、错题、阶段历史难以连续；
- 维护两套项目成本高；
- Cloudflare / GitHub Pages / 文档 / 备份都要重复维护。

判断：

- 只在当前项目已经难以扩展时才值得考虑；
- 以现状看，还没必要走到这一步。

### 4.3 路径 C：单应用多 Program 架构

做法：

- 当前项目保留为 `advanced_fluency_v1`；
- 新增 `elementary_closure_v1`；
- 共享外壳、打印、同步、批改、错题框架；
- 分离 program 的题库、缓存、画像和阶段判定。

优点：

- 不打断当前系统；
- 阶段边界清楚；
- 数据连续但不混乱；
- 未来可以继续扩第三阶段。

缺点：

- 需要先做一层程序壳抽象；
- 比单纯加题更花一点前期设计成本。

### 4.4 推荐结论

**推荐路径 C：单应用多 Program 架构。**

理由：

- 它最符合你现在的真实诉求：
  - 不动当前系统；
  - 新增下一阶段；
  - 有正式切换接口；
  - 后面还能继续扩。

## 5. 下一阶段的定位

### 5.1 建议命名

建议把下一阶段命名为：

`Mini Challenge Closure`

中文可显示为：

`小学计算收束阶段`

这个命名比“升级版”“第二套题”更清楚，因为它直接表达了目标：

- 不是简单刷更多题；
- 不是把题做难；
- 而是把小学阶段和计算有关的知识点真正收束起来。

### 5.2 阶段目标

下一阶段的目标不是“再讲新知识”，而是做到这四件事：

1. 把小学计算主干内容全部纳入一个可验证的训练框架；
2. 强化“跨表示切换”的能力：
   - 整数、小数、分数、百分数、比、比例、单位之间切换；
3. 强化“方法选择”的能力：
   - 口算、竖式、简算、通分、约分、方程、估算，什么时候用哪一种；
4. 形成“升段前稳定度”：
   - 正确率稳、
   - 易错点可解释、
   - 迁移时不塌、
   - 连续多天波动可控。

### 5.2.1 建议量化目标

为了让“收束阶段”不是只停留在概念层，建议给第二阶段一组能用于判断 readiness 的量化目标。

建议至少追踪这 4 类指标：

1. `准确率`
   - 最近 10 套训练中，核心计算域总体正确率达到目标线；
   - 高价值训练域不能长期明显低于平均线。

2. `稳定度`
   - 同一训练域连续 3 次以上不出现明显大波动；
   - 不能出现“今天会、明天塌、后天又会”的高抖动状态。

3. `迁移度`
   - 同一知识点换一种表示法后，不应持续塌陷；
   - 错题变式回炉后的重复错误率应下降。

4. `完成效率`
   - 纸面训练应能在目标时长内稳定完成；
   - 不应靠“做得特别慢”换来表面上的高正确率。

### 5.3 输出形态

下一阶段仍然保留你现在最看重的产品形态：

- 每天纸面训练；
- 系统自动生成；
- 系统记录批改；
- 根据历史动态调训；
- 优先高价值训练。

但训练页建议明确为：

- **每个学生每天仍输出 2 页题卷**
- 和当前系统一致，保留答案页、批改、错题闭环

说明：

- 这里延续当前 A/B 每个孩子 2 页的训练强度，更稳，也不需要改打印工作流的物理假设；
- 如果后续你想把“下一阶段”压缩成“总共 2 页”，可以作为单独参数，而不应作为本阶段的第一版核心约束。

## 6. 范围定义：什么属于下一阶段，什么不属于

### 6.1 纳入范围

下一阶段建议覆盖下面 8 个训练域：

1. `整数与位值收束`
   - 大数读写与比较中的位值判断
   - 多位数加减乘除
   - 借位链、退位链、末尾 0、估商

2. `小数计算收束`
   - 小数加减乘除
   - 商与积的小数点定位
   - 同倍扩大缩小
   - 小数混合运算
   - 小数估算与数量级判断

3. `分数计算收束`
   - 同分母、异分母、带分数
   - 分数乘除与连算
   - 约分、通分、交叉约分
   - 分数与整数、小数混合

4. `百分数 / 比 / 比例 / 折扣 / 税率 / 浓度 的计算化训练`
   - 重点不放在应用题文字量，而放在核心数量关系和计算链
   - 例如：百分率、折扣、增长率、比值、按比例分配、基础正反比例感知

5. `跨表示互化与桥接`
   - 分数、小数、百分数、比的互化
   - 常见基准数的快速识别
   - 单位换算与数量级桥接

6. `混合运算与策略选择`
   - 运算顺序
   - 简算策略
   - 补偿、拆分、提取公因数、凑整
   - 估算与结果合理性判断

7. `方程化与数量关系计算`
   - 用方程表达简单数量关系
   - 未知数在不同位置
   - 比例式与等量关系
   - 不是走初中代数路线，而是服务小学计算闭环

8. `速度与稳定度专项`
   - 口算簇
   - 限时短链
   - 高频易错复现
   - 低错误率维持训练

### 6.1.1 训练域与现有系统的继承关系

为了避免第二阶段看起来像完全陌生的新东西，建议明确它和现有系统的承接关系：

| 第二阶段训练域 | 与现有系统关系 | 主要来源 |
| --- | --- | --- |
| 整数与位值收束 | 直接继承并重组 | `KAI.bigSub`、`Lorik.sub`、现有乘除结构题 |
| 小数计算收束 | 直接继承并加强 | `GenKAI.decimalsMult`、`GenKAI.decimalsDiv`、`GenLorik.div` |
| 分数计算收束 | 直接继承并重组 | `GenKAI.fracEquations`、`GenLorik.fracMix` |
| 跨表示互化与桥接 | 直接继承并加强 | `getConversions`、`getLorikConv` |
| 混合运算与策略选择 | 直接继承并改组织方式 | `GenKAI.olympiad`、`GenLorik.basicMixed` |
| 方程化与数量关系计算 | 继承后重命名定位 | 现有 KAI 方程块 |
| 百分数 / 比 / 比例 / 单位换算 | 新增重点模块 | 当前系统只有部分基础，不完整 |
| 速度与稳定度专项 | 新增轻量模块 | 当前系统有基础，但没有独立 stage 设计 |

这个矩阵的价值是：

- 第二阶段主要是“基于现有系统重组和扩展”；
- 不是把已有训练全部推倒重来。

### 6.2 暂不纳入范围

下面这些内容不作为下一阶段第一版核心目标：

- 几何、测量、周长面积体积的完整体系；
- 统计与概率；
- 文字量很大的综合应用题系统；
- 初中正式代数体系；
- “奥数化”专项拓展。

如果后续要做，可以作为 `Stage 3` 或平行模块，而不是塞进这次的“计算收束阶段”。

## 7. 产品结构设计：如何不影响现有逻辑

### 7.1 总原则

不在当前生成逻辑上“硬改出第二套题”，而是把现有系统抽象成一个 `program`，再新增一个 `program`。

也就是说，架构上从：

- 一个训练系统

升级为：

- 一个训练壳 + 多个训练程序

这里的“训练壳”明确指你现有系统里已经成熟的这些部分：

- 顶部控制区
- set number 导航
- `paper-container`
- question sheet / answer sheet 渲染容器
- print sandbox
- `StorageDB`
- `GistSync`
- 错题本
- 本套批改报告
- 知识点地图

也就是说，下一阶段不应该重写这些壳层，只应该新增：

- 新的 generator
- 新的 section map
- 新的 training domain
- 新的 readiness / promotion 逻辑

### 7.2 建议的新顶层结构

建议引入：

```js
const TRAINING_PROGRAMS = {
  advanced_fluency_v1: { ... },
  elementary_closure_v1: { ... }
};
```

并增加：

- `currentProgramId`
- `program-specific generator registry`
- `program-specific profile namespace`
- `program-specific print title / section map / validation`

这样做的好处是：

- 当前 `Mini Challenge Advanced` 完全保留；
- 新阶段独立开发；
- 两套程序可以共存；
- 未来还可以继续加第三阶段。

### 7.3 UI 入口设计

在主控制区域新增一个轻量的阶段入口：

- `当前训练：Mini Challenge Advanced`
- `切换到：小学计算收束阶段`

设计原则：

- 默认仍进入当前阶段；
- 新阶段要显式进入；
- 不能因为上线新阶段而让当前用户路径发生变化。

对应到现有页面，就是在当前控制面板中新增一块非常轻的 program selector，而不是改掉现有按钮组。

必须保持不变的现有用户动作包括：

- 切换 set
- 打印 AB 四页
- 打印答案
- 打开错题本
- 打开本套批改报告
- 打开知识点地图
- 打开数据备份 / 云同步

### 7.4 数据隔离策略

下一阶段不要直接复用现有 profile 的所有字段。

建议采用：

```js
profile.programs = {
  advanced_fluency_v1: { ...existing shape... },
  elementary_closure_v1: { ...new stage shape... }
}
```

同时保留跨阶段共享摘要：

- `global_mastery_summary`
- `global_error_families`
- `promotion_history`

这样可以避免两个阶段互相污染：

- 当前阶段的弱项权重不直接扭曲下一阶段；
- 下一阶段又可以继承“孩子在哪些大域薄弱”的长期印象。

## 8. 阶段切换接口设计

### 8.1 为什么必须有正式接口

如果没有阶段切换接口，系统会出现两个问题：

1. 用户不知道什么时候算“练完当前阶段”；
2. 系统无法把当前阶段的数据变成下一阶段的初始画像。

所以需要一个真正的“毕业 + 入段”接口。

### 8.2 建议的入段条件

不建议只按“训练满一个月”切换。

建议采用：

`时间条件 + 训练表现条件`

例如：

- 已训练至少 24~30 天；
- 最近 10 套训练中：
  - 关键计算域平均正确率达到目标线；
  - 错题回炉命中后回错率下降到阈值内；
  - 高价值训练项不再持续高权重堆积；
  - 打印训练量能够稳定完成。

### 8.3 建议的毕业判定

可定义一个 `promotion readiness score`：

- `accuracyScore`
- `stabilityScore`
- `transferScore`
- `errorDecayScore`
- `completionScore`

当综合得分超过阈值时：

- 系统提示：`已具备进入小学计算收束阶段的条件`
- 用户可选择：
  - 立即切换
  - 继续留在当前阶段巩固 7 天

### 8.4 切换后的初始化

进入下一阶段时，系统要做一次“画像折算”：

- 将当前阶段的长期弱域映射到下一阶段训练域；
- 将已稳定掌握的内容直接降低基础权重；
- 将当前最不稳定的 3~5 个高价值域作为下一阶段前两周重点。

### 8.5 建议的阶段节奏

建议把第二阶段内部再细分成 3 个小周期：

1. `收口期`
   - 前 7~10 天；
   - 重点是把第一阶段画像映射进新的训练域。

2. `收束期`
   - 中间 2~3 周；
   - 重点是跨表示、策略选择、比率与单位、速度稳定度。

3. `毕业判定期`
   - 最后 5~7 天；
   - 重点减少新题，增加稳定度抽检与迁移复核。

这样更符合你说的“一个月以后进入下一阶段”的训练节奏。

## 9. 下一阶段的训练模型

### 9.1 训练原则

下一阶段不再以“单个题型覆盖”为主，而应以“能力闭环”组织。

建议采用四条并行主线：

1. `算法稳固`
   - 该竖式的会算
   - 该规则的会用

2. `表示切换`
   - 同一个数量能在不同形式之间切换

3. `方法选择`
   - 什么时候用简算、通分、约分、方程、估算

4. `稳定压错`
   - 连续多天保持低错误率，而不是偶尔一次满分

### 9.2 每日两页建议结构

建议每个学生每天 2 页，结构如下：

#### 第 1 页：算法与表示页

- A1 口算 / 速算簇
- A2 整数 / 小数 / 分数核心竖式或链式计算
- A3 互化桥接
- A4 估算与结果合理性判断

#### 第 2 页：混合与迁移页

- B1 混合运算与策略选择
- B2 百分数 / 比 / 比例 / 单位换算计算链
- B3 方程化数量关系
- B4 错题回炉 + 同类变式 + 稳定度抽检

这比当前“按学科块切 6 区”更适合下一阶段，因为它更像能力组织，而不是题型罗列。

### 9.2.1 每日两页的出题原则

第二阶段建议固定采用这 5 条出题原则：

1. `半固定骨架`
   - 每天都有稳定主结构，避免孩子题感漂移。

2. `小比例动态插针`
   - 只在少数格子里做强个性化调训，不让整卷被画像牵着走。

3. `不追求满页堆满`
   - 优先保证单题价值，而不是视觉上塞满版面。

4. `同页内保持表示切换`
   - 防止孩子一看到这一页就自动套同一种方法。

5. `回炉题不喧宾夺主`
   - 回炉与变式只占有限比例，避免整卷变成错题本复印件。

### 9.2.2 题量建议区间

建议不要一开始把题量写死，而是先给区间：

- 第 1 页：
  - 16~24 个小格，偏算法、互化、估算；
- 第 2 页：
  - 10~16 个题位，偏迁移、策略、比率、单位与稳定度抽检。

原则是：

- 第一页强调“手感密度”；
- 第二页强调“思维含金量”；
- 总体仍要适配你现在的打印密度和完成时长。

### 9.3 题量建议

下一阶段不建议盲目加题。

建议：

- 第 1 页密度略高；
- 第 2 页题量略少但每题含金量更高；
- 总体保持孩子在固定时长内能完成；
- 目标是“高价值完成率”，不是“堆满整页”。

## 10. 题库设计原则

### 10.1 不要只做“更多题”

下一阶段题库必须做到：

- 每个训练域有固定骨架；
- 每天覆盖关键误区；
- 同时允许局部个性化加权；
- 能根据错误类型给出真正有迁移价值的变式。

### 10.2 下一阶段比当前阶段新增的高价值题类

建议重点补这几类：

1. `跨表示混算`
   - 小数 + 分数
   - 百分数 -> 小数 -> 分数
   - 比值与除法之间切换

2. `单位量换算型计算`
   - 长度、面积、体积、时间、人民币等基础换算链
   - 重点是算，不是几何

3. `百分率与变化率`
   - 折扣、增减百分比、成数等

4. `按比例分配与基础比例链`
   - 以小学范围为限，不扩成初中正式比例应用题体系

5. `估算与验算`
   - 结果区间判断
   - 反向验算
   - 数量级错误拦截

6. `多表示同题变式`
   - 同一关系分别用算式、方程、比值、百分率表达

### 10.2.1 第二阶段第一批最值得上线的题类

如果要控制第一版开发量，我建议先优先做这 6 组：

1. `百分数-小数-分数三向互化`
2. `比值-除法-分数三向桥接`
3. `单位换算后的计算链`
4. `含估算和验算的合理性判断题`
5. `同一数量关系的算式 / 方程双表达`
6. `速度稳定度短链`

这 6 组的价值最高，因为：

- 它们正好补当前系统相对薄的部分；
- 又最符合“计算收束阶段”的阶段定位。

### 10.3 题型组织方式

不要继续完全按 `KAI decimals / Lorik fractions` 这类主题切。

下一阶段应引入：

- `skill family`
- `representation family`
- `operation family`
- `error family`

四个维度共同标记一道题。

这样后续才能真正做到：

- 同技能跨表示回炉；
- 同错误族跨题型迁移；
- 同一知识点的多表达训练。

## 11. 自适应策略设计

### 11.1 当前自适应机制哪些可复用

这些逻辑应直接复用：

- 弱项权重
- 遗忘拦截
- 错题本
- 同类变式
- 高价值训练优先
- 打印与批改闭环

### 11.2 下一阶段新增的自适应信号

下一阶段建议新增：

1. `representation gap`
   - 同一知识点在某种表示下总是错

2. `method selection gap`
   - 明明会算，但总选错方法

3. `stability gap`
   - 同类题一天会、三天后又掉

4. `speed-pressure gap`
   - 慢速能做对，密度稍高就出错

5. `validation gap`
   - 结果明显不合理却没有自我检查

### 11.2.1 与现有自适应信号的关系

这些新信号不应替代现有：

- `weights`
- `lastSeen`
- `errorBook`
- `domain boost`
- `high-value focus`

而应叠加在现有体系之上。

推荐做法是：

1. 保留现有一整套权重与忘记曲线逻辑；
2. 在第二阶段 profile 内新增：
   - `representationGap`
   - `methodGap`
   - `stabilityGap`
   - `speedGap`
   - `validationGap`
3. 这些信号只作用于第二阶段，不反向污染第一阶段。

### 11.3 高价值优先级建议

下一阶段的优先级不应再只是“哪里错得多练哪里”，而应强调：

- `跨表示错误` > `单一形式错误`
- `方法选择错误` > `单纯数字算错`
- `高频生活数学链错误` > `冷门技巧错误`
- `稳定性塌陷` > `偶发性失误`

这会让训练更像“升段前质量治理”。

## 12. 数据与缓存设计

### 12.1 新增的数据结构

建议在阶段切换时引入：

```js
appState = {
  currentProgramId,
  currentStageId,
  setCounterByProgram: {},
  programReadiness: {},
  promotionHistory: []
}
```

每个学生画像则变为：

```js
studentProfile = {
  programs: {
    advanced_fluency_v1: { ... },
    elementary_closure_v1: { ... }
  },
  globalMasteryMap: {},
  promotionNotes: []
}
```

### 12.1.1 第二阶段 program profile 建议字段

第二阶段画像结构建议尽量兼容现有数据形状，但增加少量 stage-specific 字段：

```js
elementary_closure_v1 = {
  weights: {},
  lastSeen: {},
  history: [],
  errorBook: {},
  representationGap: {},
  methodGap: {},
  stabilityGap: {},
  speedGap: {},
  validationGap: {},
  readinessSnapshots: []
}
```

这样做的好处是：

- 现有 `StorageDB` 逻辑比较容易复用；
- 第二阶段又能存自己真正关心的新信号。

### 12.2 缓存策略

缓存键必须带 program 维度。

例如：

```text
MathSetData_advanced_fluency_v1_103
MathSetData_elementary_closure_v1_01
```

否则两套训练会互相撞缓存。

### 12.3 错题本策略

建议：

- 阶段内错题本独立；
- 跨阶段保留一层全局错因摘要；
- 不建议把当前阶段的原题直接搬到下一阶段继续刷；
- 下一阶段更适合读入“错因摘要”而不是“题目原文”。

### 12.4 迁移与兼容策略

第二阶段设计必须考虑当前已有数据如何平滑进入新结构。

建议采取：

1. `懒迁移`
   - 第一次真正进入第二阶段时，再初始化 `programs.elementary_closure_v1`。

2. `保守迁移`
   - 第一阶段原始 `history / errorBook / weights` 保持不动；
   - 只是包进 `programs.advanced_fluency_v1` 的兼容结构。

3. `可回退`
   - 第二阶段上线后，用户随时可以切回第一阶段；
   - 两边 cache 和 profile 互不覆盖。

4. `云同步向后兼容`
   - Gist 数据结构升级后，旧数据仍能被兼容读取；
   - 不能因为第二阶段上线让老设备读档失败。

## 13. 页面与交互设计

### 13.1 首页结构

建议保持现有页面主体不动，只新增：

- 当前训练项目标签
- 训练项目切换入口
- 阶段准备度提示

### 13.2 新阶段启动页

第一次切换到下一阶段时，建议展示一页轻量说明：

- 这是“计算收束阶段”
- 目标不是更难，而是更完整、更稳
- 每天仍是 2 页
- 系统会继续记录、批改、调训

### 13.3 复盘与报告

下一阶段的报告不应只显示：

- 对 / 错

还应显示：

- 算法是否稳定
- 表示切换是否稳定
- 方法选择是否稳定
- 结果检查是否稳定

这比当前更适合“阶段毕业判断”。

### 13.4 阶段专属新页面是否需要

建议第一版不要新开完整新页面。

更推荐：

- 继续复用现有主页面；
- 继续复用同一组控制按钮；
- 只在题卷结构、答案页标题、批改报告结构上做 program-aware 区分。

这样最不打断你现在已经跑顺的使用习惯。

### 13.5 第二阶段前端设计方向

虽然第二阶段第一版不建议新开整页，但它在前端体验上仍应有清晰的“阶段感”。

建议的视觉方向是：

1. `更清楚的阶段层级`
   - 让用户一眼知道自己当前处于：
     - 第一阶段 `Advanced`
     - 还是第二阶段 `Closure`

2. `更稳定的家长操作感`
   - program 切换、阶段 readiness、打印、批改这些入口都应在现有控制面板附近完成；
   - 不要让家长为了进入第二阶段而重新学习页面路径。

3. `更轻的视觉提示，不打断纸面训练`
   - 阶段信息主要出现在控制区和少量说明卡片；
   - 学生题卷本身仍然保持考试卷风格，不要重新堆回训练标签。

4. `更明确的“阶段目标感”`
   - 第二阶段不是更难，而是更完整、更稳；
   - 这层信息应通过界面文案和 readiness 卡表达出来，而不是把题目做得花哨。

### 13.6 现有页面上的新增组件建议

建议只新增 4 个轻量组件，不动现有整体框架：

#### 组件 A：Program 切换条

位置：

- 顶部主控制面板内，靠近 set control 和打印区

作用：

- 显示当前 program
- 提供 program 切换入口

建议内容：

- 左侧：当前训练 `Mini Challenge Advanced` / `小学计算收束阶段`
- 右侧：切换按钮或下拉

设计要求：

- 它要像“工作模式切换”，不要像网页导航栏
- 视觉要明显，但不能压过打印和批改主按钮

#### 组件 B：阶段状态卡

位置：

- 主控制区内，program 切换条下方

作用：

- 显示第二阶段 readiness 或当前阶段状态
- 承担“为什么现在推荐进入 / 不推荐进入第二阶段”的解释

建议内容：

- 当前阶段名称
- 当前阶段目标一句话
- readiness 状态
- 推荐动作：
  - 继续巩固
  - 可切换第二阶段

设计要求：

- 用信息卡，不要做弹窗优先
- 状态词比百分比分数更重要，避免家长被抽象指标吓到

#### 组件 C：第二阶段首次进入说明卡

位置：

- 首次切换到第二阶段时，插入在题卷上方

作用：

- 用极短说明告诉家长和孩子：
  - 这不是“换一套更难的题”
  - 而是“计算收束阶段”

建议内容：

- 名称
- 目标
- 每天仍然 2 页
- 系统仍会继续记录、批改、调训

设计要求：

- 首次出现后可收起
- 不应在每次打开都挡住页面

#### 组件 D：第二阶段报告摘要条

位置：

- `showSetReview` 的第二阶段报告顶部

作用：

- 让报告从“错题列表”升级成“训练质量摘要”

建议内容：

- 算法稳定度
- 表示切换稳定度
- 方法选择稳定度
- 结果检查稳定度

设计要求：

- 不要做大图表第一版
- 先用简洁指标条和 1 句说明完成

### 13.7 视觉风格建议

第二阶段不建议脱离当前网站视觉另开一套设计语言。

更合适的方向是：

- 保持当前清爽、偏打印友好的基调
- 第二阶段只增加少量更“结构化”的界面元素
- 不引入强装饰、花哨图形或重卡片化布局

具体建议：

1. `颜色`
   - 第一阶段可继续用现有主色体系
   - 第二阶段建议增加一个偏深青 / 墨绿的 program accent
   - 不建议用高饱和亮色强调“升级”

2. `字体与信息层级`
   - 继续以可读性优先
   - 标题、状态、操作入口三层层级要更清楚

3. `卡片风格`
   - 控制区卡片可略增强结构边界
   - 题卷区仍然保持轻边框、弱装饰

4. `动画`
   - 只建议 program 切换时做轻微状态变化
   - 不建议在训练主界面加入明显动效

### 13.8 打印优先原则

第二阶段所有前端设计都必须服从一个更高优先级原则：

`不能影响打印`

具体包括：

- program 标识主要放控制区，不要把花哨视觉元素塞到题卷页眉
- readiness、阶段说明卡默认不进入 question-sheet 打印 DOM
- 第二阶段题卷仍然使用现有 `question-sheet` 结构
- 第二阶段任何新增提示都不能改变分页高度的不确定性

### 13.9 第二阶段题卷设计原则

第二阶段的学生题卷在视觉上应继续遵守你现在已经确立的规则：

- 像考试卷
- 少训练标签
- 少解释性文案
- 保留真正必要的数学提示

也就是说，第二阶段前端设计的重点不在“把卷子做得更花”，而在：

- 控制区更清楚
- 阶段信息更清楚
- 报告更清楚
- 纸面题卷仍然干净

## 14. 与现有代码的挂接设计

这一节的目的是把“设计”落到当前仓库真实结构上，避免后面实施时又重新解释一遍。

### 14.1 第一批需要挂接的真实入口

基于当前代码，下一阶段第一批应挂接到这些真实入口：

- `generateOrLoadSetData`
  - 现在负责按 set number 生成或读取缓存；
  - 后续要升级为按 `currentProgramId + set number` 生成。

- `StorageDB.getProfile` / `StorageDB.forceSave` / `StorageDB.saveSession`
  - 现在默认读写单层学生画像；
  - 后续要支持 program 维度。

- `renderAll` 附近的题卷拼装逻辑
  - 当前实际入口是 `window.renderPaper`;
  - 现在默认拼出 KAI / Lorik 四页和两张答案页；
  - 后续要根据当前 program 切换 section map 和标题。

- `window.printQuestionSheets`
  - 当前打印沙箱逻辑本身不用推翻；
  - 只要保证新阶段仍输出稳定的 question sheet DOM。

- `submitGrades`
  - 现在把批改回写到当前画像；
  - 后续要写回对应 program profile。

- `renderErrorBook`
  - 需要决定是“按 program 查看”还是“全局摘要 + program 分视图”。

此外，当前几个真实 UI / workflow 也要被纳入设计约束：

- `showSetReview`
  - 新阶段需要继续保留“本套批改报告”，只是报告维度要适配新的训练域。

- `showKnowledgeMap`
  - 当前已有 domain profile；
  - 下一阶段可以复用这个入口，但换成新的 domain taxonomy。

- `changeSet`
  - 后续要支持 program 维度下的 set counter，而不是只用一个全局 set number。

### 14.2 推荐的最小抽象层

建议新增几个中间函数，而不是到处写 `if currentProgramId === ...`：

```js
function getCurrentProgramId() {}
function getProgramConfig(programId) {}
function getProgramProfile(student, programId) {}
function generateProgramSetData(programId, setNumber) {}
function renderProgramSheets(programId, data) {}
```

这样可以把“下一阶段”真正变成一套配置化程序，而不是一堆条件分支。

它的价值在于：你现有 `renderPaper -> generateOrLoadSetData -> submitGrades -> StorageDB` 这条主链不需要被整体推翻，只需要在中间插入一层 program-aware 的分发。

### 14.3 建议新增的配置对象

```js
const TRAINING_PROGRAMS = {
  advanced_fluency_v1: {
    label: 'Mini Challenge Advanced',
    students: ['KAI', 'Lorik'],
    sheetsPerStudent: 2,
    generator: generateAdvancedFluencySet,
    renderer: renderAdvancedFluencySheets
  },
  elementary_closure_v1: {
    label: '小学计算收束阶段',
    students: ['KAI', 'Lorik'],
    sheetsPerStudent: 2,
    generator: generateElementaryClosureSet,
    renderer: renderElementaryClosureSheets
  }
};
```

这会是后续开发的主骨架。

### 14.4 现有代码里最该避免的实现方式

基于当前项目结构，后面实施时最该避免这几种写法：

1. 在 `renderPaper` 里直接铺大量 `if program === ...`
2. 在 `StorageDB.getProfile` 到处塞 program 分支
3. 在现有 `k_m / l_m` 这类 key 上强行混入第二阶段 key
4. 复用当前 `SET_CACHE_PREFIX` 而不带 program namespace
5. 为了第二阶段重新发明一套打印 DOM 结构

推荐原则是：

- 入口统一；
- program 配置化；
- cache 和 profile 隔离；
- 打印继续吃统一 `question-sheet / ans-sheet` 结构。

## 15. 开发方案建议

### 15.1 分阶段实施

建议按 5 个开发包推进：

#### 包 1：程序壳与阶段入口

- 增加 `TRAINING_PROGRAMS`
- 增加 program selector
- 保持当前默认逻辑不变

#### 包 2：下一阶段数据与生成骨架

- 建立 `elementary_closure_v1`
- 建立每日两页 section map
- 打通 set cache / profile namespace / print title

#### 包 3：下一阶段题库与标签体系

- 建立 8 大训练域
- 建立新的多维标签
- 建立 must-cover 骨架

#### 包 4：毕业与切换系统

- readiness score
- promotion prompt
- profile transfer

#### 包 5：验证与文档

- runtime validation
- product/dev docs
- 手工打印与训练质量验收

### 15.1.1 建议的开发里程碑

建议把 5 个开发包进一步压成 4 个里程碑：

#### 里程碑 M1：程序壳可切换

结果标准：

- 页面可选择 program；
- 默认仍是第一阶段；
- 第一阶段行为完全不变。

#### 里程碑 M2：第二阶段可生成可打印

结果标准：

- 第二阶段能生成 4 张 question sheet；
- 可打印；
- 可渲染答案页；
- 题库此时可以先不完整。

#### 里程碑 M3：第二阶段可批改可自适应

结果标准：

- 第二阶段批改可回写；
- 第二阶段错题可进入自己的 errorBook；
- readiness signal 开始积累。

#### 里程碑 M4：切换与验收闭环完成

结果标准：

- readiness 可显示；
- 阶段切换可用；
- 验证脚本补齐；
- 文档与线上版本一致。

### 15.2 为什么这样排

因为这能保证：

- 第一阶段先把“新模块能挂上去”做出来；
- 第二阶段再做核心训练内容；
- 第三阶段最后再做毕业与切换的智能判断；
- 全程不破坏现有训练主路。

## 16. 数据流与状态流

### 16.1 生成链路

建议的新链路应是：

1. 读取 `currentProgramId`
2. 读取 program config
3. 读取该 program 下当前学生画像
4. 读取该 program 下当前 set 缓存
5. 若无缓存则调用 program generator 生成
6. 渲染 question sheets
7. 渲染 answer sheets

这条链路实际上是在现有链路上演进出来的：

- 当前链路是 `renderPaper -> generateOrLoadSetData -> buildQSection/buildAnsGrid -> paper-container`
- 下一阶段建议升级成 `renderPaper -> resolveCurrentProgram -> generateProgramSetData -> renderProgramSheets`

这意味着现有 `renderPaper` 仍然可以保留为统一入口。

### 16.2 批改链路

1. Tutor 在答案页提交批改
2. `submitGrades` 收集记录
3. 写入对应 program profile
4. 更新该阶段的：
   - weights
   - lastSeen
   - errorBook
   - readiness signal
5. 如有云同步，则继续走现有 sync 链路

这也是明确参考现有系统的：

- 当前 `submitGrades` 已经和 `StorageDB.saveSession`、toast、`renderPaper`、`showSetReview` 串好了；
- 下一阶段应保留这个 UX，不要让 tutor 学一套新操作。

### 16.3 切换链路

1. 用户点击切换 program
2. 系统检查是否存在该 program 的初始化画像
3. 若无，则运行一次 transfer bootstrap
4. 设置 `currentProgramId`
5. 从该 program 的 set counter 开始渲染

### 16.4 云同步链路需要注意什么

由于现有系统已经依赖 `GistSync`，第二阶段要特别注意：

- program 切换后，push 到 gist 的仍应是一份完整总档案；
- 不建议把第二阶段拆成另一个 gist；
- 建议继续保持：
  - 一个 gist
  - 两个学生
  - 每个学生内部包含多个 program profile

## 17. 风险与错误处理

### 17.1 产品风险

- 下一阶段范围膨胀；
- 用户切换后不理解自己为什么被切换；
- 当前阶段和下一阶段的区别不够清楚。

### 17.2 技术风险

- program cache 串键；
- profile 迁移不完整；
- 打印模板在新 section map 下超高；
- 错题本视图跨 program 后变得难理解。

其中最值得提前防的，是这几个“现有系统相关风险”：

- 破坏 `打印AB四页` 这条已经稳定的主链；
- 让 `StorageDB` 的现有老数据迁移失败；
- 让 `showSetReview` 和 `showKnowledgeMap` 失去可解释性；
- 让当前默认用户一打开页面就误入下一阶段。

### 17.3 建议的防护

- 所有 cache key 强制带 programId；
- 所有 profile 读写统一通过 program accessor；
- 新阶段上线前加一次打印高度 smoke check；
- 首版错题本先做“按当前 program 查看”，全局摘要放第二阶段。

## 18. 验收标准

### 18.1 产品层

- 当前 `Mini Challenge Advanced` 行为不变；
- 可明确切换到 `小学计算收束阶段`；
- 每个学生能稳定生成每日 2 页；
- 打印与答案页不被破坏；
- 用户能看懂当前自己在哪个阶段。

### 18.2 学习层

- 下一阶段训练域覆盖“小学数与运算 + 计算型数量关系”的闭环范围；
- 每日卷不是随机拼接，而是固定覆盖关键能力；
- 错题回炉不只是回原题，而是能体现跨表示与方法迁移；
- 系统能识别“会算但不会选法”的问题；
- 可以定义清晰的阶段准备度与毕业标准。

### 18.3 技术层

- cache 按 program 隔离；
- profile 按 program 隔离；
- validation 能校验第二阶段的生成骨架；
- 不影响现有打印与同步逻辑；
- 现有用户数据可平滑迁移。

## 19. 测试与验证建议

### 19.1 运行时验证

下一阶段上线时，建议为 `scripts/validate-runtime.mjs` 新增这些检查：

- 当前 program selector 存在且默认落在 `advanced_fluency_v1`
- `elementary_closure_v1` 的 cache key 带 program namespace
- 每个学生仍渲染 2 张 question sheet
- 每张 question sheet 的 section 数和题量在允许区间内
- 新阶段题型骨架每天固定覆盖核心训练域
- 切换 program 后不会读取错的 profile

### 19.2 手工冒烟检查

- 默认打开网页，仍进入当前阶段
- 切换到下一阶段后，四张纸面正常生成
- `打印AB四页` 不出现空白页
- 答案页可正常提交
- 错题本不会混入另一 program 的原题记录

### 19.3 训练质量抽检

要人工抽检的不只是“能不能生成”，而是：

- 是否真的覆盖了整数、小数、分数、百分数、比、比例、单位、策略这几个关键域
- 是否真的出现跨表示迁移
- 是否能看出“方法选择”而不只是“数字算错”

### 19.4 第二阶段专属人工验收清单

发布前建议逐条过这份人工清单：

- 默认用户是否不会误入第二阶段
- 切到第二阶段后能否一眼看懂题卷结构
- `打印AB四页` 是否仍然稳定是 4 页有内容的题卷
- 答案页是否仍然适合家长 / 老师快速批改
- 第二阶段报告是否真的强化了“方法 / 迁移 / 稳定度”

## 20. 风险与取舍

### 20.1 最大风险

最大的风险不是代码，而是“范围失控”。

如果下一阶段一上来就想把这些全塞进去：

- 计算闭环
- 全部应用题
- 几何测量
- 初中预备

那它一定会变成一个不清楚的阶段。

### 20.2 核心取舍

这次必须坚持：

- 做“计算收束”，不做“小学数学全集”；
- 做“高价值两页”，不做“题量堆积”；
- 做“阶段接口”，不做“把旧逻辑揉烂重来”；
- 做“程序共存”，不做“替换当前系统”。

## 21. 推荐结论

我的建议非常明确：

1. **值得做下一阶段**
   - 而且产品价值很高；

2. **可以把下一阶段定义为“小学计算收束阶段”**
   - 覆盖小学 `数与运算` 主干和计算型 `数量关系`；

3. **不建议现在重构当前系统**
   - 应该把它作为 `advanced_fluency_v1` 保留下来；

4. **下一阶段必须有正式切换接口**
   - 包括 readiness、promotion、profile transfer；

5. **第一版仍保持每天 2 页 / 每个学生**
   - 与现有打印和使用习惯兼容。

## 22. 待确认点

为了让后续实施更贴近你的节奏，这里有 3 个值得你拍板的点：

1. 下一阶段默认是否仍保持 `KAI / Lorik` 双学生双卷结构
2. “一个月后切换”是偏硬性时间门槛，还是“时间 + readiness”双条件
3. 下一阶段第一版要不要把 `百分数 / 比 / 比例 / 单位换算` 全部纳入，还是先做 `整数 / 小数 / 分数 / 策略 / 方程`

这 3 个点不影响这份设计文档成立，但会影响后续实施顺序。

## 22.1 当前这版文档的推荐默认答案

如果先不额外拉长讨论，我给出的默认建议是：

1. `继续保持 KAI / Lorik 双学生双卷结构`
   - 这样和你现有打印、批改、存档主链完全兼容。

2. `切换条件用 时间 + readiness 双条件`
   - 保留你说的“一个月以后”，同时避免机械切换。

3. `第一版就纳入 百分数 / 比 / 比例 / 单位换算`
   - 但只做计算链，不扩成完整应用题系统。
